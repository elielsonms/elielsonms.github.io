# Repository Guidance

## Project Purpose
- This repository builds and publishes a personal portfolio site plus a downloadable PDF resume.

## Source Of Truth
- The portfolio data source is `datasource.yaml`.
- Prefer YAML over JSON for authoring because multiline descriptions are easier to maintain.
- Be careful with YAML strings containing `:` inside lists; quote them to avoid accidental map parsing.

## Build Flow
- Preferred local build command: `npm run build`
- This command must generate:
  - `dist/index.html`
  - `dist/Elielson-Machado-Silva.pdf`
  - `dist/styles/main.css`
  - `dist/scripts/email-reveal.js`

## Rendering Architecture
- Web HTML is generated by:
  - `scripts/generate-html.js`
  - `scripts/render-portfolio.js`
- The Node side reads YAML through:
  - `scripts/print_data_json.py`
- PDF is generated directly in Python by:
  - `scripts/generate_pdf.py`
- PDF generation uses `reportlab`.
- Do not reintroduce HTML-to-PDF generation unless explicitly requested.

## Python Execution
- Python commands should prefer `.venv/bin/python3` when `.venv` exists.
- The helper script for this is:
  - `scripts/run-python.sh`

## Deployment
- GitHub Actions workflow:
  - `.github/workflows/static.yml`
- The workflow should stay aligned with local development and use `npm run build`.
- `dist/` is generated output and should not be versioned.

## Current Web Header Decisions
- GitHub and Resume PDF links should appear discreetly below `.header-role`.
- WhatsApp belongs in the `header-meta` block.
- There should be no separate Download section below the main content.

## PDF Expectations
- The PDF should stay visually simple and compact, suitable for printing.
- Prefer a straightforward resume layout over a highly styled design.
- Aim for 1-2 pages when possible, but do not sacrifice clarity just to force the page count.
- GitHub and Portfolio links belong side by side below the role line in the PDF header.

## Editing Preferences
- Preserve the current YAML + Node + Python architecture unless explicitly asked to change it.
- Favor small, direct changes over broad refactors.
- When changing the data format or build flow, keep the local command and CI workflow in sync.
